# CLAUDE.md

## 프로젝트 개요

이 저장소는 AutoDraft 시스템의 문서 수집(ingestion) 및 색인(indexing) 서비스이다.

이 서비스의 역할은 Google Drive에 업로드된 문서를 검색 가능한 지식으로 변환하는 것이다.  
시스템은 새로 추가되거나 수정된 PDF 파일을 감지하고, 파일을 다운로드한 뒤 파싱, 청킹, 임베딩, 벡터 저장소 적재를 수행하며, 별도의 상태 저장소에 처리 상태를 기록한다.

이 저장소는 문서 수집과 색인에 책임이 있다.  
챗봇의 최종 답변 생성은 이 저장소의 책임 범위가 아니다.

---

## 이 저장소의 목적

이 프로젝트의 목적은 기존의 수동 문서 적재 흐름을 자동화하는 것이다.

기존에는 개발자가 로컬 폴더에 파일을 넣고 직접 파이프라인을 실행해야 했다.  
이 저장소는 사용자가 지정된 Google Drive 위치에 파일을 업로드하면, 해당 문서가 자동으로 처리되어 이후 챗봇 검색에 활용될 수 있도록 만드는 것을 목표로 한다.

중요한 품질 목표는 다음과 같다.

- 안정적인 문서 수집
- 추적 가능한 처리 상태
- 중복 없는 재실행 가능성
- 명확한 문서 생명주기 관리

---

## 저장소 구조

- `ingest_service/` — 문서 수집 파이프라인의 핵심 모듈
- `shared/` — 공통 설정과 데이터 모델
- `migrations/` — 상태 저장소 스키마 및 마이그레이션
- `ops/` — 운영 및 배포 관련 자산
- `Dockerfile` — 컨테이너 빌드 정의
- `requirements.txt` — Python 의존성 정의

---

## 핵심 처리 흐름

문서 수집 파이프라인의 기본 흐름은 아래와 같다.

1. Google Drive에서 변경된 파일을 감지한다.
2. 파일 메타데이터와 실제 파일 내용을 다운로드한다.
3. PDF를 파싱하여 정규화된 텍스트 또는 구조화된 결과를 만든다.
4. 검색 친화적인 단위로 청크를 생성한다.
5. 각 청크에 대해 임베딩을 생성한다.
6. 벡터 저장소에 upsert 한다.
7. 문서 처리 상태와 색인 메타데이터를 상태 저장소에 기록한다.

## 주요 모듈 안내

아래 파일들은 시스템을 이해하거나 수정할 때 먼저 확인해야 하는 핵심 진입점이다.

- `ingest_service/drive_client.py` — Google Drive API 접근
- `ingest_service/change_tracker.py` — 변경 감지 및 동기화 기준점 관리
- `ingest_service/downloader.py` — 파일 다운로드 처리
- `ingest_service/parser_adapter.py` — 파서 계층과의 연결 지점
- `ingest_service/chunker.py` — 검색용 청크 생성
- `ingest_service/embedder.py` — 임베딩 생성
- `ingest_service/vector_store.py` — 벡터 저장소 쓰기 및 갱신
- `ingest_service/state_store.py` — 문서 상태 및 생명주기 저장
- `ingest_service/worker.py` — 전체 파이프라인 오케스트레이션
- `shared/config.py` — 런타임 설정
- `shared/models.py` — 공통 스키마 및 데이터 구조

## 시스템 경계

아래 경계는 유지되어야 한다.

- 이 저장소는 문서 수집, 색인, 상태 추적을 담당한다.
- 이 저장소는 챗봇의 최종 응답 생성 로직을 담당하지 않는다.
- 벡터 저장소는 문서 처리 상태의 단일 진실 공급원(source of truth)이 아니다.
- 문서의 처리 상태와 생명주기 정보는 벡터 저장소 밖에서도 명시적으로 조회 가능해야 한다.

---

## 설계 원칙

작은 단위의 명확한 수정과 조합 가능한 구조를 우선한다.

Google Drive 접근, 파싱, 청킹, 임베딩, 상태 저장, 벡터 적재를 하나의 거대한 파일로 합치지 않는다.  
모듈 경계를 유지하고 각 단계의 책임을 분리한다.

문서의 식별자와 버전 의미를 명확히 유지한다.  
재처리 여부는 파일명 같은 불안정한 기준보다, file ID, modified time, revision, 또는 이에 준하는 명확한 기준에 따라 판단해야 한다.

멱등성(idempotency)을 우선한다.  
같은 파일 버전에 대해 수집 작업이 여러 번 실행되어도 중복된 색인 결과가 생기지 않아야 한다.

실패는 숨기지 않는다.  
오류를 조용히 무시하지 말고, 실패 상태와 원인을 추적 가능하게 남겨야 한다.

검색 결과의 추적성과 재색인을 위해, 각 청크에는 충분한 메타데이터가 저장되어야 한다.

## 이 저장소에서 작업하는 방법

작업을 시작할 때 먼저 변경 대상이 어느 영역에 속하는지 판단한다.

- Drive 동기화 및 파일 감지
- 파일 다운로드
- 파싱
- 청킹
- 임베딩
- 벡터 색인
- 문서 상태 추적
- 설정 또는 스키마

수집 동작을 변경할 때는 아래 영향 범위를 함께 확인한다.

- 문서 생명주기 상태
- 벡터 메타데이터 구조
- 재처리 기준
- 재시도 및 실패 처리 방식

확신이 없을 때는 기존 계약(contract)을 유지하고, 필요한 가정은 코드 주석이나 관련 문서에 명시한다.

## 좋은 변경의 기준

이 저장소에서 좋은 변경은 다음을 만족한다.

- 문서 수집의 안정성을 높인다.
- 처리 상태를 더 명확하게 만든다.
- 재실행과 재색인을 더 안전하게 만든다.
- 코드 구조를 단순하게 유지한다.
- 챗봇 응답 생성 같은 다른 책임 범위까지 불필요하게 확장하지 않는다.
